# GalWriter AI 工业化上线 Sprint / Milestone 计划

> 目标：把当前“功能基本完整”的产品，推进到可长期交付、可回归、可观测、可维护的工业化发布状态。

## 1. 目标定义

本计划面向正式上线前的产品硬化，不是继续堆功能。最终目标是让 GalWriter AI 达到以下状态：

- 核心流程稳定：新建、打开、编辑、保存、导入导出、Playtest、视频/Web 导出都能可靠运行。
- 回归可控：每次改动都能被自动化检查和关键路径测试覆盖。
- 问题可追踪：线上/现场能定位故障，而不是只靠用户截图和控制台。
- 数据可恢复：项目数据、自动保存、迁移、兼容旧版本都可验证。
- 发布可回滚：版本升级、配置升级、数据升级都有退出路径。

## 2. 工业标准完成定义

达到“工业标准”的最低门槛建议定义为：

- 关键路径自动化覆盖率达到可维护水平，且每次发布前必须通过。
- 至少有一套端到端回归集，覆盖保存、打开、导入、导出、恢复、关闭。
- 所有核心状态变更都有明确的单向数据流或可解释同步策略。
- 所有持久化数据都有版本策略、迁移策略、异常回退策略。
- 错误不只打印到控制台，至少具备统一错误处理和日志采集入口。
- 发布流程可重复，构建产物、版本号、变更说明、回滚方案齐备。

## 3. Milestone 总览

### Milestone A：稳定基线

目标：先把“会误报、会乱亮、会丢状态”的问题压住。

交付物：

- 统一项目快照口径，保存状态与自动保存状态一致。
- 打开项目后不误判为脏状态。
- 保存、重命名、关闭、恢复的状态流清晰。
- 关键项目状态切换增加最小化诊断日志开关。

验收标准：

- 打开已有项目后，保存按钮默认不高亮。
- 未修改内容时，不触发 autosave。
- 修改后、保存后、重开后状态一致。

### Milestone B：测试底座

目标：让回归不靠人工记忆。

交付物：

- 单元测试框架落地。
- 核心领域逻辑测试：项目序列化、反序列化、迁移、保存名计算、自动保存判定。
- 集成测试：项目打开/保存/导入/导出基础流。
- E2E 测试：最少覆盖一条完整编辑闭环。

验收标准：

- CI 上可以自动跑测试。
- 核心变更不需要人工逐项点页面确认。

### Milestone C：可靠性与恢复

目标：把数据安全和恢复能力补齐。

交付物：

- autosave 恢复策略完善。
- 保存失败、导出失败、读取损坏项目时有明确回退提示。
- 项目数据版本号与迁移策略落地。
- 本地缓存、草稿、临时文件清理策略明确。

验收标准：

- 任意异常中断后，用户能恢复到最近可用状态。
- 旧版本项目在新版本中可打开，必要时可自动迁移。

### Milestone D：可观测与可诊断

目标：问题出现后，能快速判断是哪里坏了。

交付物：

- 统一错误边界或全局错误上报入口。
- 关键业务操作埋点：打开、保存、导入、导出、恢复、崩溃。
- 本地日志分级：info / warn / error。
- 诊断包或日志导出能力。

验收标准：

- 线上问题能给出项目上下文、版本、操作阶段、错误栈。
- 不再依赖“用户发一张图猜问题”。

### Milestone E：发布工程化

目标：让发布变成可重复流程，而不是手工仪式。

交付物：

- CI/CD 流水线。
- 版本号、变更说明、构建产物签名或校验。
- 灰度发布或测试版通道。
- 回滚预案。

验收标准：

- 任意一次发版都可追溯构建来源。
- 出问题时可以明确回退到哪一版。

## 4. Sprint 计划

### Sprint 1：状态同步与保存链路硬化

重点工作：

- 统一快照生成和比较口径。
- 修正打开项目时的 dirty 状态和保存按钮高亮。
- 审核 `projectTitle`、显示名、文件名的职责边界。
- 收敛 autosave 与手动保存的基线同步逻辑。

产出：

- 一个稳定的项目打开/保存状态流。
- 一个最小回归清单。

### Sprint 2：关键路径测试

重点工作：

- 给 project serializer、local persistence、autosave 增加测试。
- 写保存/恢复/导入导出的关键用例。
- 补一条编辑器主流程 E2E。

产出：

- CI 可跑的测试套件。
- 关键问题可被测试捕获。

### Sprint 3：异常处理与恢复

重点工作：

- 统一错误提示与恢复提示。
- 项目损坏、缺字段、旧结构、导出失败的回退处理。
- 修复“部分成功、部分失败”场景下的状态残留。

产出：

- 可恢复的异常处理链。
- 更少的静默失败。

### Sprint 4：数据迁移与兼容性

重点工作：

- 明确数据版本策略。
- 为本地存储、项目 ZIP、导入 JSON 增加迁移测试。
- 旧版本字段、缺省字段、废弃字段的兼容处理。

产出：

- 新旧版本互操作能力。
- 数据变更可控。

### Sprint 5：可观测性与排障

重点工作：

- 为保存、导入、导出、恢复、启动添加统一日志上下文。
- 增加错误分级和诊断入口。
- 定义本地排障步骤和最小日志集。

产出：

- 团队能快速定位线上/现场问题。

### Sprint 6：发布工程化与上线门禁

重点工作：

- CI/CD 完整化。
- 发布前门禁检查。
- 版本说明、回滚说明、验收清单固化。

产出：

- 一个可以重复执行的正式发布流程。

## 5. 建议排期

- 每个 sprint 建议 1 到 2 周。
- 如果当前人力较少，优先完成 Sprint 1、2、3。
- 如果目标是正式对外长期运行，Sprint 4 到 6 不建议跳过。

## 6. 优先级原则

优先级从高到低建议是：

1. 数据正确性和不丢数据
2. 保存 / 打开 / 恢复稳定性
3. 自动化测试
4. 错误可观测性
5. 迁移和兼容性
6. 发布工程化

## 7. 发布前验收清单

- 打开已有项目不误亮保存按钮。
- 不修改内容时关闭应用不会触发未保存提示。
- 修改内容后保存、重开、恢复都符合预期。
- 旧项目、损坏项目、缺字段项目都能给出清晰反馈。
- 导出 ZIP、导出 Web、导出视频都能成功或明确失败。
- 至少一套自动化回归在 CI 中稳定通过。
- 版本号、发布说明、回滚方案已准备。

## 8. 备注

这份计划的核心不是“再做更多功能”，而是把当前产品从“功能完整”推进到“可持续交付”。  
如果后续要继续扩展 AI 能力、导出能力或协作能力，最好也建立在这套工业化底座之上。
